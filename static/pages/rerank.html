Re-ranking

Retrieve cheap with semantic, then re-order with a smarter model. The 2-stage trick behind most production RAG.

try this

  1. Type a query and run. We retrieve the top-15 by semantic similarity, then ask an LLM to score each (query, doc) pair 0–10.
  2. The big table is "all candidates scored". The 🟢 / 🔴 highlights show which docs moved up or down vs the cheap retriever.
  3. Top-5 "before" vs "after" sits at the top.
  4. Notice: the LLM often promotes docs that semantically share meaning but didn't have the perfect embedding distance — and demotes off-topic near-neighbors.
Think of it like: a hiring funnel. A 5-minute resume scan narrows 1,000 applicants to 15, then a real interviewer (more expensive, much smarter) picks the top 5. Same pattern: fast & dumb first, slow & sharp second.
why two stages instead of one?

An LLM is too slow/expensive to score every doc in your corpus. A vector index can score millions instantly but is imprecise. The trick: retrieve cheap top-K, then use the LLM as a precise judge on a small candidate set. Real systems use small cross-encoder models (rerankers) for this; we use the chat model itself to demonstrate the concept.
